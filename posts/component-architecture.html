Component Architecture for Design Systems

June 20, 2023

Components are the building blocks of modern user interfaces and the heart of any design system. But creating a component architecture that's flexible, maintainable, and scalable requires careful planning and thoughtful design decisions. In this post, we'll explore strategies for structuring components in a way that serves both designers and developers.

The Anatomy of a Well-Designed Component

Before diving into architecture, let's consider what makes a component "good" in the context of a design system:

  • Reusable: Can be used in multiple contexts without modification
  • Composable: Can be combined with other components to create more complex UIs
  • Configurable: Adaptable to different use cases through props/parameters
  • Accessible: Follows accessibility best practices by default
  • Consistent: Aligns with the overall design language
  • Documented: Has clear usage guidelines and examples

Component Hierarchy: Atomic Design

One popular approach to component architecture is Atomic Design, developed by Brad Frost. This methodology breaks down components into five distinct levels:

1. Atoms

The smallest, indivisible UI elements:

  • Buttons
  • Input fields
  • Icons
  • Typography elements

2. Molecules

Simple groups of atoms functioning together:

  • Search form (input + button)
  • Navigation item (icon + text)
  • Form field (label + input + error message)

3. Organisms

More complex UI sections composed of molecules and atoms:

  • Header (logo + navigation + search)
  • Product card (image + title + price + button)
  • Comment section (avatar + username + text + actions)

4. Templates

Page-level structures that arrange organisms:

  • Article layout
  • Product detail page
  • Dashboard layout

5. Pages

Specific instances of templates with real content.

This hierarchical approach helps teams think systematically about component relationships and promotes reuse at every level.

Component Types and Patterns

Beyond the atomic hierarchy, it's useful to categorize components by their functional role in the system:

Primitive Components

These are the low-level building blocks that directly implement design tokens:

  • Box/Container: For layout and spacing
  • Text: For typography styles
  • Icon: For visual elements

Composite Components

These combine primitives into more complex, reusable patterns:

  • Card: A container with specific styling and layout
  • Modal: A dialog with backdrop, content area, and actions
  • Tabs: A navigation pattern with tab list and content panels

Layout Components

These manage the arrangement of other components:

  • Grid: For two-dimensional layouts
  • Stack: For one-dimensional layouts (vertical or horizontal)
  • Split: For dividing space between components

Component API Design

How you define the interface for your components is crucial for their usability and longevity. Here are some principles to consider:

Props vs. Composition

There are two main approaches to component configuration:

  • Props-based: Configuration through properties 
    <Button size="large" variant="primary" icon="download">
  Download
</Button>
  • Composition-based: Configuration through child components 
    <Button>
  <Icon name="download" />
  <Text>Download</Text>
</Button>

Most design systems use a combination of both approaches, with props for simple configuration and composition for more complex scenarios.

Variants vs. Props

When designing component APIs, you'll need to decide between predefined variants and individual props:

  • Variant approach: 
    <Button variant="primary" />
<Button variant="secondary" />
<Button variant="danger" />
  • Props approach: 
    <Button primary />
<Button secondary />
<Button danger />

The variant approach is generally more maintainable as it encapsulates related properties and makes it easier to add new variants without changing the API.

Naming Conventions

Consistent naming is essential for a coherent component library:

  • Use clear, descriptive names that indicate the component's purpose
  • Be consistent with prefix/suffix patterns (e.g., all form components prefixed with "Form")
  • Use standard prop names across components (e.g., "size" instead of mixing "size" and "dimension")

Implementation Strategies

The Base Component Pattern

Many design systems use a "base component" pattern to promote consistency and reduce duplication:

// Base component with core functionality
const BaseButton = ({ children, ...props }) => (
  <button {...props}>{children}</button>
);

// Specialized variants built on the base
const PrimaryButton = (props) => (
  <BaseButton className="btn-primary" {...props} />
);

const SecondaryButton = (props) => (
  <BaseButton className="btn-secondary" {...props} />
);

Composition vs. Inheritance

In component-based architectures, composition is generally preferred over inheritance:

  • Composition: Building complex components by combining simpler ones
  • Inheritance: Extending component classes/functions to add or modify behavior

Composition leads to more flexible, maintainable components that are easier to reason about and test.

Styling Approaches

The way you style components can significantly impact your architecture:

  • CSS Modules: Scoped CSS files imported into components
  • CSS-in-JS: JavaScript libraries that generate CSS at runtime
  • Utility Classes: Composing styles from small, single-purpose classes
  • Custom Properties: Using CSS variables for theming and configuration

Each approach has trade-offs in terms of performance, developer experience, and flexibility.

Documentation and Testing

A robust component architecture includes not just the components themselves, but also:

Component Documentation

  • Usage examples
  • Prop/API documentation
  • Design guidelines
  • Accessibility considerations
  • Interactive playgrounds (e.g., Storybook)

Testing Strategy

  • Unit tests for component logic
  • Visual regression tests for appearance
  • Accessibility tests
  • Integration tests for component interactions

Conclusion

A well-designed component architecture is the backbone of a successful design system. By thoughtfully structuring your components with clear hierarchies, consistent APIs, and flexible implementation patterns, you can create a system that scales with your organization's needs.

Remember that component architecture is not just a technical concern—it's a collaboration between design and development that shapes how teams build products. The best architectures emerge from close partnership between these disciplines and evolve based on real-world usage and feedback.
